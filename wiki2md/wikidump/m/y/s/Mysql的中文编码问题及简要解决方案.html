Mysql的中文编码问题及简要解决方案

Wikipedia，自由的百科全书

从前使用的mysql版本是4.1的，默认配置，运行书中的中文例子时，正确显示没有问题。再加上平时动手写的页面大多是英文的，这就回避开了mysql的中文编码问题。在自己手动配置了一次mysql之后，种种问题都暴露出来了，通过自己的解决，感觉收获了不少东西。看来只有像cliff说的那样多遇到错误，才能不断进步。


状况1：将mysql设置为latin1（ISO-8859-1），所有书中 (Tomcat与Java Web) 例子显示都是正常的，因为有convert函数将ISO-8859-1转码为GB2312。

分析：如果采用默认的latin1，存入数据库时会将GB2312编码的二进制数当作ISO-8859-1编码的（数据库不知道你存入的数据是何种编码，只是一些二进制数），字符内码不变，怎样存进去就怎样取出来。这样在显示时，如果不加以转换的话，真值一直都没有变，只是认为是ISO-8859-1的编码（本来应该是GB2312的编码），显示出来的是按照ISO-8859-1编码的字符串（在JSP页面中从ResultSet中获得的是String――字符内码的表现形式），这样一定会是乱码。用convert函数可以解决这个问题： 

public String convert(String s){
    try{
    	 return new String(s.getBytes("ISO-8859-1"),"GB2312");
    }catch(Exception e){return null;}
}

原理是将得到的字符串用ISO-8859-1的编码格式转换回二进制的数组，再用GB2312的格式转换回字符串，这样就能够显示正常了。


状况2：将mysql设置为GB2312，所有书中 (Tomcat与Java Web) 例子显示都是正常的，此时不再用convert函数将ISO-8859-1转码为GB2312。

分析：因为数据库，页面的编码都是GB2312，不存在转换问题，也就不会遇到中文显示乱码的问题了。


状况3：为什么会有“?”号

分析：种语言之间的转换是通过 Unicode来完成的。假设有两种不同的语言A和B，转换的步骤为：先把A转化为Unicode，再把Unicode转化为B。 两个方向转换都有可能得到错误的结果：

  • 当从某语言向Unicode转化时，如果在某语言中没有该字符，得到的将是Unicode的代码“\uffffd”（“\u”表示是Unicode编码）。
  • 从Unicode向某语言转化时，如果某语言没有对应的字符，则得到的是“0x3f”（“?”）。这就是“?”的由来。
