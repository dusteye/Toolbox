一般规矩


1．涉及字符 

26个英文字母（区分大小写）和0－9这十个自然数，加下划线共63个字符（注释除外）。

2．在小于30字符的情况下，数据表名称使用英文单词全称进行命名,全称表示他是什么



3．前缀一律使用2~4个字符的缩写代替单词(三个字符优先，四个字符的不用缩写)，前缀表示他属于哪一部分 

例如：
FBlog      fb_
表          tb_
视图        vi_
存储过程    sp_
函数        fn_

用户表      user_
链接表      link_
评论表      com_

比如说他是link表，然后他属于group部分的，就用 gro_links

4.所有表和字段的名称中，用下划线分隔每个单词。


5.表示计数的用num,不用times.以免与“时间”相混淆。

表：

1. 

character set(字符集)统一为：utf8
collation(字符间比较的方法) 统一为：utf8_general_ci

2.表名采用复数形式 

例如：users

3.某一个表xxx的扩展表采用xxx_advanced的形式 

例如：users表的扩展表名称为users_advanced

4.连接表使用所连接的两个表的复数形式，中间用下划线分隔表示 

例如：users_links

5.相同功能模块的几个表添加相同的前缀 

例如：与会议相关的表统一添加“conf_”前缀：conf_notices（普通表），conf_conferences_members（连接表）

字段：


1.每个表中只设一个主键id，表示该表的id，作为外键的id要添加前缀，前缀名为引用的表名的单数形式。 

例如：group表中用外键user_id引用表users。

特殊字段（添加中）

所有储存url的字段采用同一的长度150

视图命名规范

我们约定，字段由前缀和实际名字组成，中间用下划线连接。前缀：使用小写字母vi，表示视图。因此，合法的视图名类似如下。 

vi_User
vi_UserInfo


[编辑]

存储过程命名规范


我们约定，字段由前缀和实际名字加操作名字组成，中间用下划线连接。前缀：使用小写字母sp，表示存储过程。操作名字：Insert|Delelte|Update|Caculate|Confirm 例如： 

sp_User_Insert

sql语句规范 

我们约定，所有sql关键词全部大写，比如SELECT,UPDATE,FROM,ORDER,BY等。

数据库设计文档规范

所有数据库设计要写成文档，文档以模块化形式表达。大致格式如下： 

'-------------------------------------------
'    表名：    tbUser_Info
'    作者：    Yezi(叶子)
'    日期：    2004-12-17
'    版本：    1.0
'    描述：    保存用户资料
'    具体内容：
'    UserId        int，自动增量        用户代码
'    UserName    char(12)        用户名字
'    ......
'--------------------------------------------

管理规则

数据库在统一的一台机器上建立，由专人负责管理，个人不能擅自修改，如果需要修改的话要联系数据库管理员。现在由_____负责数据库的管理。

数据库修改规则


1.数据库的表格不得擅自修改，包括各个字段的名称、大小、及是否为空，因为这对于操作数据库有关，又是可能因为一个小小的改变，使我们在操作的时候发生好多异常而难以找到这样微小的错误。

2.对于数据库中以后需要添加相应的字段，我们大家需要商量。 具体为：

（1）增加此字段的理由，大家各抒己见，这就和设计表格时一样需要通过大家的一致同意。

（2）对于要增加字段的表格，在那些文件中运用到，需要大家必须列出来，以便在改变数据库表格时对相应的文件加以改变，尤其是配置文件。

（3）对于需要修改或者删除的字段流程同（2）

（4）大家在自己的机器上所建立的数据库要和服务器上的一致，包括数据库字符编码等。以避免在两处同样的操作而产生不同的结果（错误）。

3.对于数据库的修改可以按照下面的进行，将相应的内容添加到每个条目下面 

修改时间 
原始表格 
修改后表格 
修改的字段及理由 
和此表格相应的文件列表
