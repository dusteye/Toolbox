图形显示过程简述


如图<未上传>（光栅显示系统结构）所示，cpu或gpu在系统内存或显存中进行图形相关的计算，最终将会形成一幅和我们所看到的屏幕相对应的逻辑图象。该图象通过PCI总线或AGP接口送到帧缓存中。此时这副图象以连续内存的方式（线形）存在于帧缓存中。屏幕上每一个象素对应内存中的一位，一个字节或若干个字节（这取决于颜色深度）。DAC或视频控制器（图2<未上传>）里的光栅扫面发生器的扫描信号和地址发生器的地址引用同步生成。使屏幕象素与内存象素一一对应。而且视频控制器还根据从帧缓冲里提取的象素信息（象素索引值）在自己的颜色查找表（CLUT）中找出真正要显示的颜色值，例如对rgb输出来说，就是rgb三种颜色的强度值。最后通过vga或dvi接口将象素信息和扫面控制信号输出到显示器，我们就可以看到图象了。

Framebuffer设备抽象

在linux中，对显示设备的管理和控制是通过fb设备实现的。我们分别可以通过“文件”/dev/fb*,/sys/class/graphics/fb* ,/proc/bus/pci/devices（假设我们的显示设备是个pci设备）得到关于该物理设备的相关信息。例如通过/dev/fb*就可以将帧缓存映射到用户空间的一块内存，对这块内存的读写马上就可以从屏幕上看出来。再例如利用sysfs或/proc/bus/pci/device 就可以到pci设备配置空间的部分信息，因此利用这些信息我们可以在用户空间编写驱动程序。确实也有人这样做，在DirectFB中就是通过这种方式得到 硬件信息，然后在此基础上做硬件加速。 另外fb设备提供了大量的ioctl操作，包括改变显示器的显示模式，改变屏幕的显示位置，设置颜色查找表 等（可参考fbset）。因此我们可以关注fb设备，以及它在系统（内核）中的表示，建立和提供的接口来看看它是怎样 控制显示设备的。

设备模型

fb设备就是一个（或多个）字符设备（29），设备路径是/dev/fb*，它以文件的方式为用户提供访问和控制底层显示设备的统一接口。这个我们将进入内核源代码/driver/video/fbmem.c 中，看看内核是怎样建立fb设备的。但是首先我们需要明白fbmem并不直接和硬件打交道，那是fb硬件驱动完成的，fbmem只是根据fb驱动提供的信息在系统中实现fb设备。因此我们可以把framebuffer设备分为两层：设备驱动和设备抽象。1.2试图说明设备抽象，1.3将介绍fb设备驱动。
我们拿来抽象的资本是内核对一个或多个显示设备的描述，其数据结构是struct fb_info(在 $LINUXSOURCE/include/linux/fb.h 中定义)，具体清单如下：

struct fb_info {
	int node;
	int flags;
	struct fb_var_screeninfo var;	/* 显示设备可变信息：主要包括显示模式，颜色信息等*/
	struct fb_fix_screeninfo fix;	/* 显示设备固定信息:内存映射地址，大小，是否支持加速等 */
	struct fb_monspecs monspecs;	/* Current Monitor specs */
	struct work_struct queue;	/* Framebuffer event queue */
	struct fb_pixmap pixmap;	/* Image hardware mapper */
	struct fb_pixmap sprite;	/* Cursor hardware mapper */
	struct fb_cmap cmap;		/* 颜色映射表：我们直接提供的颜色表示格式不能代表具体的颜色       
                                         * 值，fb设备利用它在CLUT中检索最终的颜色值 。
                                         */
	struct list_head modelist;      /* 该设备有不通的显示模式 */
	struct fb_videomode *mode;	/* current mode */
	struct fb_ops *fbops;           /*  类似file_operations 的函数表，提供对显示设备最基本的操作
                                         *  也可以提供简单的硬件加速功能。
	struct device *device;
	struct class_device *class_device; /* sysfs per device attrs */
#ifdef CONFIG_FB_TILEBLITTING
	struct fb_tile_ops *tileops;    /* Tile Blitting */
#endif
	char __iomem *screen_base;	/*  帧缓存的内核虚拟地址  */
	unsigned long screen_size;	/*  显存大小（ms不应该叫screen_size） */ 
	void *pseudo_palette;		/* Fake palette of 16 colors */ 
#define FBINFO_STATE_RUNNING	0
#define FBINFO_STATE_SUSPENDED	1
	u32 state;			/* Hardware state i.e suspend */
	void *fbcon_par;                /* fbcon use-only private area */
	/* From here on everything is device dependent */
	void *par;	
};

各个设备的fb_info结构在内核中以两个导出的符号记录 

struct fb_info *registered_fb[FB_MAX];
int num_registered_fb;

他们的填充由驱动完成。 而fb设备的建立在fbmem_init()中实现的，他是该抽象层初始化执行的函数，以module_init或subsys_initcall方式执行。下面以注释的方式说明： 

fbmem_init(void)
{
	create_proc_read_entry("fb", 0, NULL, fbmem_read_proc, NULL);

	devfs_mk_dir("fb");
        /*
         * 这在devfs文件系统中为fb设备建立目录，但这里还没有对应具体设备，给目录下的设备通过下文的
         * register_framebuffer()建立。

		printk("unable to get major %d for fb devs\n", FB_MAJOR);

          * 它会在内核中建立一个cdev结构来表示改字符设备，并将cdev链入系统的字符设备链表进行维护。

	fb_class = class_create(THIS_MODULE, "graphics");
	if (IS_ERR(fb_class)) {
		printk(KERN_WARNING "Unable to create fb class; errno = %ld\n", PTR_ERR(fb_class));
		fb_class = NULL;
	}

          * 这在sysfs系统中建立graphic目录，返回的fb_class将被下文的register_framebuffer利用，在          
          * graphic目录下建立不同fb设备的“类”。可以查看/sys/class/graphic 。
          * 
	return 0;
}

通过fbmem_init可以看出，它为具体的设备的建立提供基础（cdev,sysfs class），针对具体的设备是由具体的驱动自己调用register_framebuffer()来做的。register_framebffer()也是一个导出的符号，因此在内核中是可见的：

这样我们就可以通过固定的文件接口来访问fb设备了：设备号相同的一“类”设备对应相同的file_operations ，就有不同此设备号的同一类设备就有不通的inode结构， 

static struct file_operations fb_fops = {
	.owner =	THIS_MODULE,
	.read =		fb_read,
	.write =	fb_write,
	.ioctl =	fb_ioctl,
#ifdef CONFIG_COMPAT
	.compat_ioctl = fb_compat_ioctl,
#endif
	.mmap =		fb_mmap,
	.open =		fb_open,
	.release =	fb_release,
#ifdef HAVE_ARCH_FB_UNMAPPED_AREA
	.get_unmapped_area = get_fb_unmapped_area,
#endif
}
