探测设备

pci设备刚加电时，设备自身将初始化配置空间里的一些寄存器，如DEVICE_ID,VENDER_ID,REVISION_ID等，同时用相应的标志位来填充COMMAND寄存器，来回应/吸收预到达此设备的一些总线信号：此设备还未配置好,拒绝访问。然后该设备等待bios或os来配置。这里的配置主要指给此设备的存储区间及io空间分配物理地址（总线地址）。所以从cpu或os的角度来讲，需要先找到（模到）改设备。

PCI总线和PCI设备组成了一个树形结构，树根是HOST-PCI桥，HOST-PCI桥下面就是主总线，所以主总线一摸就到。linux将0号总线分给主pci总线后，就开始顺藤模瓜，而且是深度优先。只要有了这个设备(主总线)，并建立相应的数据结构（pci_root_buses）。就可以在此基础上进行探测了，用linux的话说就是

/* Go find them,Rover*/
for(devfn = 0;defn<0x100;devfn +=8)

{

     dev0.devfn=devfn;

     pci_scan_slot(&dev0)；

}

其中devfn为八位宽的“逻辑设备号”，就是总线编号中的8：15位，分别代表了设备号(5位)和功能号（3位）。可见每条总线上可以有（0x100/8)个设备。详细代码课参考pci_do_scan_bus()； 这样每探测到一个设备就分配就建立一个pci_dev结构，将改结构挂到该设备所在的总线上(DEV->bus)，进而进行相关设置。而探测到一个PCI-PCI桥，linux建立一个pci_bus结构来代表改桥下游总线，并将最小的一个未分配总线号分配给改总线。进而探测改总线上的设备或桥。

我们可以通过/sys/bus/pci/目录可以了解到系统的pci结构。知道那个bus上有那些设备，那个设备挂到了那个bus上。

在以上过程中，linux是通过HOST-PCI上的两个固定io端口0XCF8~0XCFF（分别地址寄存器和数据寄存器）来访问设备的，linux将设备的总线编号及相应的逻辑设备号放在地址寄存器中（组成的地址不是总线地址），然后同过数据寄存器向目标设备写/读数据。这些操作被封装在PCI_OP_READ和PCI_OP_WRITE中，进而gcc在编译是进行字符替换（用"##"标记）成pci_read_config_*()等。通过这些读写操作和linux枚举设备，linux可以建立其相应的数据结构来描述系统的pci结构了，这些结构是这样的，pci_root_buses对列代表HOST-PCI桥下的主总线，每个pci_bus下的devices是一个队列，代表此bus上的所有设备，pci_bus下的children队列代表该bus上的PCI-PCI桥下的次层总线。在这些结构的基础上，开始进行实际的配置。

读取设备信息

linux对配置空间的读写是通过pci_read(write)_config_*来读写的，它封装了对对io端口0XCF8~0XCFB 和0XCFC-0XCFF的读写。以pci_read_config_word为例: 

 int pci_read_config_word(struct pci_dev *dev,int pos,u16*value);

dev代表读写对象，pos为寄存器在配置空间的偏移在pci.h中有定义，是对pci设备配置空间的描述，设备返回值保存在value中。 通过这一族“函数”的读写（pci_scan_device()），linux将设备空间的基本信息读到该设备的pci_dev中。但是总线地址的分配还未进行，需要根据系统的所有io资源进行分配。

[编辑]

分配地址

总线地址的分配比较复杂，linux需要对许多中不同的情况进行处理，例如bios已经为某些设备分配了地址等，其过程还再理解中。但其目的是从系统的整个io资源中分配设备所需的地址大小给设备，这是通过将分配给设备的起始地址写入设备的基址寄存器实现的。同时将该设备资源情况记录到pci_dev->resources 中包括其实地址和大小，而resource是一个数组，resource[0]记录了io地址资源情况，resource[1]内存地址空间资源情况。这样，系统可以通过分配给设备的总线地址来直接访问该设备的io空间和内存空间了。

寻找驱动

内核一方面可以被动的通过用户命令insmod加载，另一方面当内核检测到某硬件而硬件驱动尚未安装，也需要主动的去寻找相关的驱动。无论是那种情况都需要模块（驱动）与设备的对应（不是一一对应）。通过MODULE_DEVICE_TABLE()宏将一个pci_device_id 数组导出到内核文件/lib/modules/KERNEL_VERSION/modules.pcimap中可以实现他们的对应，以intelfb来说他的pci_device_id 是这样的：

static struct pci_device_id intelfb_pci_table[] __devinitdata = { 
{ PCI_VENDOR_ID_INTEL, PCI_DEVICE_ID_INTEL_830M, PCI_ANY_ID, PCI_ANY_ID,     
         PCI_CLASS_DISPLAY_VGA << 8, INTELFB_CLASS_MASK, INTEL_830M },
{ PCI_VENDOR_ID_INTEL, PCI_DEVICE_ID_INTEL_845G, PCI_ANY_ID, PCI_ANY_ID, 
        PCI_CLASS_DISPLAY_VGA << 8, INTELFB_CLASS_MASK, INTEL_845G },
/*...*/
{ 0, }
};

该结构说明了那些设备可以使用这个驱动。而利用这些信息，linux可以找到该设备对应模块的名称及其他信息。进而就可以加载驱动了。

[编辑]

加载驱动

驱动加载时要进行设备的参数设置，并需要将设备的总线地址映射到内核空间，来避免直接使用硬件地址进行操作。以intelfb为例，这是在intelfb_pci_register（）中实现的： 

static int _devinit intelfb_pci_register(struct pci_dev *pdev, const struct pci_device_id *ent)；

而参数pdev是代表设备的pci_dev结构,在pci初始化是已经填充了其中的相关字段，如resource[]就记录了该设备的总线地址。包括控制、状态等寄存器的内存映射空间，和帧缓冲区的地址。分别在pdev->resource[0]和pdev->resource[1]中记录。intelfb用(struct intelfb_info *dinfo)来记录设备的相关信息。驱动分别将两项地址读入到dinfo中 

dinfo->aperture.physical = pci_resource_start(pdev, aperture_bar);
dinfo->aperture.size     = pci_resource_len(pdev, aperture_bar);
dinfo->mmio_base_phys    = pci_resource_start(pdev, mmio_bar);

然后用request_mem_region做一些检查，是否有别的设备使用该地址，并将记录该资源的使用情况，这样内核可以统一管理这些io资源。 以后intelfb还要在此地址范围内得到更详细的地址使用信息，例如ring buffer,cursor等，并将这些信息记录到dinfo中。 这时设备的地址使用细节知道以后，就通过ioremap将地址映射到内核空间： 

dinfo->aperture.virtual = (u8 __iomem *)ioremap_nocache (dinfo->aperture.physical, ((offset +  dinfo->fb.offset) << 12) + dinfo->fb.size);
dinfo->mmio_base = (u8 __iomem *)ioremap_nocache(dinfo->mmio_base_phys ， INTEL_REG_SIZE);

昨晚映射以后，对设备的读写就可以基于内核虚拟地址了，如对设备io接口寄存器的读写可以同过INREG/OUTREG来实现： 

#define INREG(addr) readl((u32 __iomem *)(dinfo->mmio_base + (addr)))
#define OUTREG(addr, val)    writeb((val),(u32 __iomem *)(dinfo->mmio_base + (addr)))
